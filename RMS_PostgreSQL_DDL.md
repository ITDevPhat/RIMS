# RMS — PostgreSQL DDL (converted từ SQL Server)

## Kết luận trước

Chuyển đổi 1:1 về mặt logic, không giữ nguyên literal syntax. Các quyết định kiến trúc chính:

| Vấn đề SQL Server | Quyết định PostgreSQL | Lý do |
|---|---|---|
| `ROWVERSION` (auto binary version) | `row_version BIGINT DEFAULT 1` + trigger `BEFORE UPDATE` tăng dần | Postgres không có kiểu tương đương thật; dùng `xmin` hệ thống là anti-pattern (không bền qua VACUUM FULL/dump-restore) |
| `NVARCHAR(MAX)` chứa JSON + `CHECK (ISJSON(...)=1)` | `JSONB` | Loại bỏ hẳn constraint kiểm tra thủ công, được index hoá, query native |
| `DATETIME2(0)` | `TIMESTAMPTZ` | Tránh bug timezone-naive khi hệ thống scale ra nhiều region; nếu bắt buộc naive UTC, đổi thành `TIMESTAMP(0)` |
| `IDENTITY(1,1)` | `GENERATED ALWAYS AS IDENTITY` | Chuẩn SQL:2003, tránh phụ thuộc `SERIAL` (legacy, không có ràng buộc identity thật) |
| `BIT` | `BOOLEAN` | Không có kiểu BIT tương đương ngữ nghĩa flag |
| `OUTER APPLY TOP 1` | `LEFT JOIN LATERAL ... LIMIT 1` | Cú pháp chuẩn Postgres cho correlated subquery |
| Computed column `PERSISTED` | `GENERATED ALWAYS AS (...) STORED` | Tương đương trực tiếp |
| SQL Server stored procedures | `PROCEDURE` (không trả dữ liệu) / `FUNCTION` (có trả `SCOPE_IDENTITY()`) | Postgres tách biệt PROCEDURE (side-effect, dùng `CALL`) và FUNCTION (trả giá trị) |
| `GO` batch separator | Bỏ, dùng `;` + statement riêng | Không tồn tại trong Postgres |

**Rủi ro cần biết trước khi chạy:**
- `GENERATED ALWAYS AS IDENTITY` chặn insert có chỉ định ID thủ công trừ khi dùng `OVERRIDING SYSTEM VALUE`. Nếu cần migrate data cũ kèm ID gốc, đổi sang `GENERATED BY DEFAULT AS IDENTITY`.
- Toàn bộ định danh (table/column) giữ nguyên `snake_case` — không cần quote, an toàn cho Postgres case-folding.
- `row_version` không tự tăng khi `INSERT`, chỉ tăng khi `UPDATE` (đúng hành vi `ROWVERSION` gốc).
- Chưa có statement `pgcrypto`/`uuid-ossp` vì schema gốc dùng toàn `BIGINT IDENTITY`, không có UUID.

### 1.1 Common trigger function (row_version)

```sql
CREATE OR REPLACE FUNCTION audit.fn_bump_row_version()
RETURNS TRIGGER
LANGUAGE plpgsql
AS $$
BEGIN
    NEW.row_version := OLD.row_version + 1;
    RETURN NEW;
END;
$$;
```

CREATE OR REPLACE VIEW research.v_research_project_overview AS
SELECT
    p.project_id,
    p.project_code,
    p.project_title,
    d.department_name AS lead_department_name,
    pi.full_name AS principal_investigator_name,
    COALESCE(s.sponsor_name, p.sponsor_name_text) AS sponsor_name,
    p.protocol_number,
    p.protocol_version,
    p.ethics_status,
    p.ethics_expiry_date,
    p.planned_start_date,
    p.planned_end_date,
    p.progress_percent,
    p.current_phase_name,
    p.project_status,
    p.health_status,
    p.risk_level,
    nearest.next_due_date,
    nearest.next_due_title,
    CASE
        WHEN p.ethics_expiry_date IS NOT NULL AND p.ethics_expiry_date < CURRENT_DATE THEN true
        ELSE false
    END AS is_ethics_expired,
    CASE
        WHEN p.ethics_expiry_date IS NOT NULL
             AND p.ethics_expiry_date BETWEEN CURRENT_DATE AND (CURRENT_DATE + INTERVAL '30 day')
        THEN true
        ELSE false
    END AS is_ethics_expiring_soon
FROM research.research_projects p
LEFT JOIN ref.departments d ON p.lead_department_id = d.department_id
LEFT JOIN auth.users pi ON p.principal_investigator_id = pi.user_id
LEFT JOIN research.sponsors s ON p.sponsor_id = s.sponsor_id
LEFT JOIN LATERAL (
    SELECT
        dl.due_date AS next_due_date,
        dl.deadline_title AS next_due_title
    FROM research.project_deadlines dl
    WHERE dl.project_id = p.project_id
      AND dl.deleted_at IS NULL
      AND dl.deadline_status IN ('open','overdue')
    ORDER BY dl.due_date ASC
    LIMIT 1
) nearest ON true
WHERE p.deleted_at IS NULL;


CREATE OR REPLACE VIEW training.v_training_monthly_summary AS
SELECT
    event_year,
    event_month,
    SUM(CASE WHEN plan_type = 'planned' THEN 1 ELSE 0 END) AS planned_count,
    SUM(CASE WHEN plan_type = 'additional' THEN 1 ELSE 0 END) AS additional_count,
    SUM(CASE WHEN event_status = 'completed' THEN 1 ELSE 0 END) AS actual_count,
    SUM(CASE WHEN plan_type IN ('planned','additional') THEN 1 ELSE 0 END) AS total_plan_count,
    GREATEST(
        SUM(CASE WHEN plan_type IN ('planned','additional') THEN 1 ELSE 0 END)
        - SUM(CASE WHEN event_status = 'completed' THEN 1 ELSE 0 END),
        0
    ) AS not_completed_count,
    CAST(
        CASE
            WHEN SUM(CASE WHEN plan_type IN ('planned','additional') THEN 1 ELSE 0 END) = 0 THEN 0
            ELSE SUM(CASE WHEN event_status = 'completed' THEN 1.0 ELSE 0 END)
                 / SUM(CASE WHEN plan_type IN ('planned','additional') THEN 1.0 ELSE 0 END) * 100
        END AS NUMERIC(5,2)
    ) AS completion_rate
FROM training.training_events
WHERE deleted_at IS NULL
  AND is_active = true
GROUP BY event_year, event_month;

CREATE OR REPLACE PROCEDURE notify.p_mark_notification_read(
    p_notification_id BIGINT,
    p_user_id         BIGINT
)
LANGUAGE plpgsql
AS $$
BEGIN
    UPDATE notify.notification_recipients
    SET is_read = true,
        read_at = COALESCE(read_at, now())
    WHERE notification_id = p_notification_id
      AND user_id = p_user_id;
END;
$$;


CREATE OR REPLACE PROCEDURE notify.p_mark_all_notifications_read(
    p_user_id BIGINT
)
LANGUAGE plpgsql
AS $$
BEGIN
    UPDATE notify.notification_recipients
    SET is_read = true,
        read_at = COALESCE(read_at, now())
    WHERE user_id = p_user_id
      AND is_read = false;
END;
$$;


-- Trả về notification_id (tương đương SCOPE_IDENTITY) -> FUNCTION, không phải PROCEDURE
CREATE OR REPLACE FUNCTION notify.p_create_notification(
    p_notification_type   VARCHAR(100),
    p_category             VARCHAR(100),
    p_title                 VARCHAR(500),
    p_message                 TEXT,
    p_priority_level            VARCHAR(50),
    p_related_entity_type          VARCHAR(100),
    p_related_entity_id              BIGINT,
    p_related_entity_code              VARCHAR(100),
    p_recipient_user_id                  BIGINT
)
RETURNS BIGINT
LANGUAGE plpgsql
AS $$
DECLARE
    v_notification_id BIGINT;
BEGIN
    INSERT INTO notify.notifications (
        notification_type, category, title, message, priority_level,
        related_entity_type, related_entity_id, related_entity_code
    )
    VALUES (
        p_notification_type, p_category, p_title, p_message, p_priority_level,
        p_related_entity_type, p_related_entity_id, p_related_entity_code
    )
    RETURNING notification_id INTO v_notification_id;

    INSERT INTO notify.notification_recipients (notification_id, user_id, delivered_in_app_at)
    VALUES (v_notification_id, p_recipient_user_id, now());

    RETURN v_notification_id;
END;
$$;
-- Gọi: SELECT notify.p_create_notification('deadline','deadline','Sắp hết hạn','...', 'high', 'research_project', 12, 'RP-2026-01', 5);

## Rủi ro & hành động tiếp theo

1. **Chưa test trên PostgreSQL thật** — chạy `psql -f RMS_PostgreSQL_DDL.sql` trên instance sạch (Docker `postgres:16`) trước khi merge vào pipeline AWS Glue → SQL Server hiện tại của bạn. Lưu ý: file này là schema **đích PostgreSQL**, không liên quan trực tiếp tới medallion SQL Server đang build cho ICON Phase 1 — nếu đây là hệ thống khác (RMS), giữ tách biệt hoàn toàn khỏi Glue Workflow hiện tại.
2. **`GENERATED ALWAYS AS IDENTITY`** chặn insert ID thủ công cho seed/migration dữ liệu cũ — nếu cần nạp lại ID gốc từ SQL Server, đổi toàn bộ sang `GENERATED BY DEFAULT AS IDENTITY` trước khi chạy migration một lần, rồi có thể giữ nguyên `ALWAYS` cho vận hành production sau đó.
3. **`row_version` qua trigger** tăng thêm 1 round-trip nhỏ mỗi UPDATE — chấp nhận được ở OLTP quy mô này; nếu sau này cần optimistic concurrency ở scale lớn hơn, cân nhắc thay bằng `pg_catalog.pg_current_xact_id()`-based hoặc chuyển hẳn sang application-level ETag.
4. **TIMESTAMPTZ vs TIMESTAMP**: đã chọn TZ-aware. Nếu toàn hệ thống chỉ chạy 1 timezone cố định (VN, UTC+7) và ứng dụng luôn set `session.timezone` nhất quán, có thể đổi về `TIMESTAMP(0)` để giảm overhead — nhưng đây là premature optimization, giữ TIMESTAMPTZ là lựa chọn an toàn hơn về lâu dài.
